# 【转载】谈研发和技术积累的好文：20年工作经验[转]

<div style="text-align: center;"   >

</div><p style="text-align: center;"   ><span style=""   >马宁伟<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp; </span></span><span style=""   >我是一个有<span lang="EN-US"   >10</span>年电子产品研发经验的工程师和<span lang="EN-US"   >10</span>年<span lang="EN-US"   >IT</span>知名公司研发中心管理经验的技术管理者。世上好的管理理念可能归纳起来就那么<span lang="EN-US"   >1~2</span>百条，也都好理解，难的是怎么适当地运用在特定的环境中。下面的文章共<span lang="EN-US"   >18</span>篇，是我<span lang="EN-US"   >20</span>年工作中的片段，也是我在研发及管理中的实际体验和感悟。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1. </span><span style=""   >技术启蒙 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1977</span><span style=""   >年的<span lang="EN-US"   >5</span>月我进了宿迁无线电厂。今天已极少有人知道这家厂， 但当时这家厂还是有些名气的，因为当时设计新产品技术队伍中有在文革中遭解散的南京无线电工业学校的校长，教导主任，系主任，各科教师。其中有中国的第一代电化学专家叶厚禺，第一代无人驾驶康拜因<span lang="EN-US"   >(</span>拖拉机<span lang="EN-US"   >)</span>的无线电控制设计者赵骥，第一代雷达设计师赵柏林，第一代无线电专家洪瑞楫，第一代飞机机械师龚维蒸，还有靠“自学成才”的<span lang="EN-US"   >60</span>年代即是华电工程师的张世昌。这些人才，多是因政治运动，先被排挤到学校再被下放到宿迁。当时工厂设计的晶体管图示仪和集成电路测试仪以及微波测试仪器产品畅销全国。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >龚维蒸老师原为系主任，当时已有<span lang="EN-US"   >50</span>岁，头发花白，是南工（现南京东南大学）双专业的<span lang="EN-US"   >50</span>年毕业的本科生，也是中国第一本《无线电设备结构设计》教材的编写者。在工厂是结构设计室的主任，当时我们有<span lang="EN-US"   >6~7</span>个高中毕业生跟他学模具设计和仪器结构设计。每天半天学习理论和做作业，半天工作。当时的工厂主产品<span lang="EN-US"   ><span style=""   >&nbsp; </span>JT</span>－<span lang="EN-US"   >1</span>、<span lang="EN-US"   >JT</span>－<span lang="EN-US"   >3</span>晶体管图示仪畅销，每台成本几千块，每台销售价<span lang="EN-US"   >7</span>万<span lang="EN-US"   >5</span>千块。市场竞争也不激烈。<span lang="EN-US"   ></span></span></p>



<p><span style=""   >龚维蒸老师主要给我们上<span lang="EN-US"   >4</span>种课，《机械制图》，《材料力学》，《冷冲模设计》《无线电设备结构设计》。他讲课深入浅出，循循善诱，结合实例，总是鼓励人，从不批评人，也从没看过他发过脾气，真正的敦厚长者。当时我们基础实在是太差，他讲完一点，总是会问：听懂了么？因为我的好问和勤奋，使我很快在几个人中脱颖而出。<span lang="EN-US"   >2</span>年多的上课使我打下了良好的无线电设备结构设计理论基础，我当时还花半个月的工资买了铁摩辛柯的《材料力学》硬啃了很长时间，一直到读了大学才知道，龚老师的课选的是多么准确和有效，使我们一点弯路没走。我的一个大学老师跟我们说的是：学好《机械制图》，《材料力学》，给你半个工程师。对一个电子产品结构设计工程师来说，真是金玉良言。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >我的另一个师傅是张世昌，因为工厂要自制一套<span lang="EN-US"   >4</span>开制版机（用来做<span lang="EN-US"   >PCB</span>的胶片）我被抽调到这个产品设计小组跟他学做结构设计，实际上主要是制图，将他的设计思想，用图纸表达出来。第一天他先让我削只铅笔，结果是他不满意，他削了只让我看，用中华<span lang="EN-US"   >2H</span>铅笔，笔芯削成<span lang="EN-US"   >20mm</span>长的圆锥过渡，铅芯要露出<span lang="EN-US"   >5</span>－<span lang="EN-US"   ><span style=""   >&nbsp;
</span>6mm</span>长，再用细沙纸磨成圆锥型，笔尖直径<span lang="EN-US"   >0.2mm</span>。他用丁字尺划了一条直线，一边划，铅笔一边转，铅笔与图板倾斜成<span lang="EN-US"   >60</span>度。他说，这样笔尖不会一下被磨秃。然后，他让我写<span lang="EN-US"   >0~9</span>的阿拉伯数字。我写完后，他说，你数字写的不错，我推荐你去财务科，看我脸涨的通红，他说“<span lang="EN-US"   >8</span>”字要这样写，我看着他用<span lang="EN-US"   >l</span>了<span lang="EN-US"   > 4 </span>笔，每笔<span lang="EN-US"   >4</span>个半圆写成了标准的仿宋体“<span lang="EN-US"   >8</span>”。就这样我跟着他一丝不苟的开始了制图工作。他教会我选择三视图的技巧，运用局部剖视图和重复剖面图的表达方法，特别是强调要精确，简洁，完整的表达零件，不能少表达一个要素，也不能多划一个视图。“让水平低的人也能看懂你的图，好的工程图纸就是一幅艺术划”
这些设计理念，深深地影响了我。他极富创意，教会我巧用机械原理，实现制版机结构和传动的设计。一个个想法从图纸变为实体零件，也使我渐渐地喜欢上了这个工作。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >事实上，张世昌老师还是一个多才多艺的人，他做的古筝，被淮阴地区剧团长期使用。他甚至教我们自己制做小提琴，告诉我们要用什么木料和特别的剖料方式。这个问题在我后来带的多个研发团队中，作为有奖问答，几乎无人答出。他给我也是给那个小城留下最深的影响是，当时有一出大热的话剧叫《于无声处》各地都在排演。话剧两个多小时，要反映的情节是<span lang="EN-US"   >24</span>小时。张世昌老师用一座钟改变传动比，加上硬纸板做成大钟面和表针，用缝纫机做车床加工精密零件。钟做成后，挂在舞台上，成功的反映了剧情时间的变化。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >张世昌老师家居图书馆，他太太是<span lang="EN-US"   >50</span>年代的北大图书馆系毕业生。当时是县城图书馆馆员，星期天我去借书时，常看他一家都在图书馆院中读书，他看的杂志，书籍面非常广，比如象《化石》，《乐器制作》。我从小就养成了阅读的习惯，但限于文学类，进工厂后，因工作需要也常借技术书看。受他的影响，我开始读《地理知识》《航空知识》《科学画报》《化石》《考古》《摘译》等五花八门的杂志。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1977</span><span style=""   >－<span lang="EN-US"   >1978</span>是我学习技术的最幸福时光。那样的学习工作条件，可能今天的研究生和一流企业也不能具备，工厂环境在当时也称得上是花园工厂，有良师，有一个好工作，有技术可学，衣食无忧，没有什么竞争，还有一个自己心仪的女孩是同事，刚刚从生活的阴影中走出来的我，真的好象是来到了理想中的乌托邦。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >我是<span lang="EN-US"   >1970</span>年随父母从南京下放到宿迁的，<span lang="EN-US"   >1974</span>年在宿迁中学高中毕业，<span lang="EN-US"   >1977</span>年才进工厂，岁月蹉跎。一位与我有同样经历的小学同学对我说，人要有“一技之长”，才能在社会上立足。所以，从进工厂起，我就有了个理想，奋斗<span lang="EN-US"   >10</span>年，成为工程师。 <span lang="EN-US"   ></span></span></p><p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >2.</span><span style=""   >三年寒窗 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >1978</span><span style=""   >年我参加了高考，可惜没考上，主要原因是当时我的工作已不错了，没有全身心投入复习，现在看来是鼠目寸光。原本准备来年再考，可突然来了读大学的机会，那就是中央广播电视大学开办并冬季招生，<span lang="EN-US"   >1979</span>年<span lang="EN-US"   >2</span>月开学。开始并没打算读，可是读电大有一个好处是，可以带薪上学。就这样，通过简单考试，我开始了读电大。此时国家落实知识分子政策，原南无复校，老师们陆续回宁。工厂新产品设计需要人，所以读电大的<span lang="EN-US"   >3</span>年我是一边读书，一边工作。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >大学的三个要素：名教授，图书馆，实验室。实际上电大仅具备了第一个。当时授课的大多是清华，北大老师。电大也成就一批名师，象教<span lang="EN-US"   >Basic</span>语言的谭浩强，教英文的郑培蒂被媒体评为全国最多学生的老师，他们走到哪里，都被优先照顾，因为到处都有他们的学生。电大是全国统一教材，统一上课，通过电视同步传到各地。由于还没有线电视，用无线接收。电视差转台常出故障，刮风下雨，天线摇摆，转播效果更差，屏幕上往往是雪花飘飘。一节课耽误了，就无法弥补，全靠自学。电大采取的是宽进严出，考试是全国统一试卷，考试地点是相对集中。我们是在地区市的淮阴。完全没有普通大学老师学生面对面的种种便利。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们那个班开始有<span lang="EN-US"   >50</span>多人，一学期下来就不见了一半，一年后还剩<span lang="EN-US"   >15</span>人，到毕业，只有<span lang="EN-US"   >8</span>个人。我因是一边读书，一边工作，每天只睡<span lang="EN-US"   >4~5</span>个小时，整整<span lang="EN-US"   >3</span>年几乎天天如此，。为什么有那么大的恒心？说起来，是因为工厂里自己心仪的女孩跟一个驻军的团政委儿子好上了，我觉得主要是自己的经济条件不够好，因此为了出人头地，发愤读书。实际上，现在想想不可思议，当时就没有跟人家表白过，也许追一下，就是另一种结果。<span lang="EN-US"   >3</span>年下来，人瘦成<span lang="EN-US"   >97</span>斤，而且以后<span lang="EN-US"   >10</span>年如此。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我的毕业设计是“<span lang="EN-US"   >RL</span>－<span lang="EN-US"   >2</span>微波漏能结构设计”那也是我们工厂实际研究设计的一个新产品。用来探测微波设备的微波泻漏能量。微波设备包括雷达，电视发射台等，超过<span lang="EN-US"   >30mw/cm3</span>的微波对人体是有害的。在毕业设计答辩上，我从探测原理讲起，结合挂图介绍了主要器件传感器，又介绍了为将传感器上获得的讯号能在表头上显示出来，需放大电路和转换电路，这就是电路设计工程师的工作。另一方面，为了让电路板能放在合适的位置，也为了电路不受干扰，又为了传感器要有合适的形状便于手握，又因为是移动中使用的产品，特别要考虑防振设计，而这些就是电子产品的结构设计。我还重点介绍了产品中需要的一种材料：它即要能够被制成异型。又要能耐高温不变形，还要对微波无反射。为找到这种材料（氮化硼）所做的研究过程。我的报告受到了内行和外行的好评。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我之所以要比较详细的说这件事，是象这样真正的原创性的研究和设计是非常锻炼人的，也非常有挑战性，在我后来的<span lang="EN-US"   >20</span>年研发工作这样的项目不多。事实上在很多<span lang="EN-US"   >IT</span>公司的研发部门也是如此，我见到的一家国内大公司的名校毕业生，津津有味的在做整机级测试，没人告诉他们，这不能算是研发。我的一个感觉是这些年来国内很多<span lang="EN-US"   >IT</span>大公司研发做的越来越浅，远不如<span lang="EN-US"   >80~90</span>年代的企业。所以，整体上我们工业技术可能与发达国家比现在离得更远。当然有些公司除外象华为，中兴等。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >3.</span><span style=""   >十年磨一剑 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >龚维蒸老师看我想真学技术，就对我说，要想学透一行，要花十年工夫。先用<span lang="EN-US"   >3</span>年时间，将这一行需要的基础理论知识掌握；再用<span lang="EN-US"   >3</span>年时间将这一行的书，文章全部找来读，了解这门技术的来龙去脉和当前的发展状况；再用<span lang="EN-US"   >3</span>年时间选择其中薄弱分支进行研究，这样你就可以有所成就。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >1984</span><span style=""   >年我回南京进了熊猫集团东方无线电厂，工厂主要产品是收录机和组合音响。我先是在工艺科结构室，主要的工作是：在设计阶段审查设计部门的结构设计图纸工艺性；在生产阶段承接设计部门的技术，写工艺流程卡（作业指导书）和处理生产线的技术问题。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >就现在看，后一个工作任务就是现在很多公司里工程部的职责。而前一个工作任务在大多数公司没有部门承担，这也是很多新产品样机做的可以，当批量生产时，问题就百出的原因之一。一个通则是，把技术问题解决在前端，将大大降低新产品失败的风险和成本，这是技术管理者要关注和研究的重要课题。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >半年后发生了两件对我有影响的事。第一件事是描图员忙不过来，我去客串帮忙，在描一张图时，顺便改掉了一个明显的设计表达错误。按理描图员是不应做任何改动的，所以这引起了那位设计师的不满，好在负责结构设计的副所长就在旁边，拿过去一看，说改得对，就这样了。并且问我，你怎么会知道要这样改？我说，我是做过结构设计的。第二件事是我审查一个新产品的一套结构设计图的工艺性，正巧就是这位副所长设计的。我找出来其中十几个制图和设计不当问题。我看完后，随口跟旁边的同事说了一下，谁知这信息迅速传到了他耳中，而且也不知怎么传的。据说他听后是暴跳如雷，说我的图纸还会问题，他一个小年青，搞的不得了。我的室主任说，这是个不能碰的人，因为又是交大毕业的，又是副所长，这个时候，我的倔强性格也出来了，说：白纸黑字在那呢，大家都可以看啊。结果却是，下午当面沟通时，他一一修正了问题。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >因为这两件事，由设计所长提出，将我调到了设计所。我回到了结构设计的本行，此时我实际上已有了<span lang="EN-US"   >6</span>～<span lang="EN-US"   >7</span>年的冷冲模和仪器结构设计经验。仪器主要用金属薄板做结构件；而音响的结构件主要是塑料件，塑料件的设计对我来说是个新课题，除此以外，结构设计的内容是相通的。我买了能够买到的所有塑料材料和零件设计的书。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >塑料因为可以设计成复杂零件，这特别考验一个人的三维空间想象能力。业界一直有两类设计师，一类是先设计个大概轮廓，再做一个手板，根据手板，再完善细节设计。另一类是完全想好细节，并用图纸完整表达，再去做手板，这种做法，改动会很小。但非常考验设计者的功力。我是一直坚持后者的做法。同理，做一个整机结构设计，首先要划装配图，没有精细的装配图就匆匆划零件图也注定会反复甚至翻船。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我对设计的定义就是能够将设想和计算通过图纸将其表达出来，并可以成功的做成产品。对设计师来说，关键能力是还没有实物，但这物体已在脑中想象成形。而不是依葫芦划瓢。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >调来设计所后，也是新手但有相当才气的丁石藤和我分别独立担任电路和结构设计师，设计高性能的双卡收录机<span lang="EN-US"   >8400</span>。这是个全新产品，全套结构设计图包括装配图，零部件图，我一人<span lang="EN-US"   >40</span>天完成， 开模后一次试模成功。电路也如期设计调试完成，整机试投产后也一次成功，没有发现大问题，故试产后一天都没停就又投入了<span lang="EN-US"   >6K</span>的批量。做到这个状况，任何一个做过设计的人都知道这是很难达到的一个技术水准，同时这里面还有很多的协调工作，当然还有运气，比如模具厂，<span lang="EN-US"   > PCB</span>厂没有出问题。因为成功的设计了这个产品，也因为领导和同事的认可，我被任命为结构设计室主任。这在那个论资排辈的年代和国企环境是很破格的事情。我所在的设计室基本上都是有十几年工作经验的设计师。而我的搭档也成了设计所长<span lang="EN-US"   >------</span>丁石藤是我迄今为止见到的少数几个杰出人才之一，他最历害的就是思维清晰而简洁，一语中的，具备超常人的智商，面对技术问题他几乎不会出错。而我的不出错是靠我的勤奋和专心，我在做设计时经常半夜惊醒，发现一个计算错误。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我这时才知道，龚，张两位老师教我的专业知识，技术和自己三年苦读加工作，使我已积累了相当的专业功力。连我自己也奇怪，我对塑料零件设计并不熟悉，但遇到问题我总能迅速找到解决方法，我没有什么问题需要问我的同事，相反我倒是经常帮他们的忙。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >后来我接触过很多的工程师，发现他们虽然在<span lang="EN-US"   >R&amp;D</span>部门工作了<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>年，但并没有掌握他们那个专业的知识和技能，更糟糕的是他们不知道那个专业应该学些什么。一方面我为自己感到庆幸，一方面我也得出一个结论，刚刚工作的<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>年对新大学生太重要了，没有一个好的师傅指导，会走多少弯路，甚至于就此碌碌无为。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我在结构设计室主任的位上一坐<span lang="EN-US"   >10</span>年，虽然<span lang="EN-US"   >2</span>年后我就就兼职副所长，八年后又做了所长。<span lang="EN-US"   > 8400</span>产品在生产车间量产时，在一张检验桌旁，检验科长问我，产品成功了，拿多少奖金就满意了，我说奖金不在乎，我希望能生产<span lang="EN-US"   >5</span>万台，成为畅销产品。那是真话，那个年代的工程师，更看重的是名声。事实上由于是定位在高端的产品，销路并不好。所以厂里在分析了市场后，决定大幅度削减附加功能，仅保留主要功能，从结构上仅换面板，其余沿用原来的零部件。这大大快了设计进度。产品投产后命名为熊猫<span lang="EN-US"   >2200</span>双卡收录机。就是这个产品上市后，因供不应求，包括熊猫集团本部的工厂一共<span lang="EN-US"   >4</span>家厂生产，整整畅销<span lang="EN-US"   >3</span>年，共产销<span lang="EN-US"   >120</span>万台。为工厂创下巨额利润。也创下全国音响行业单机种销量的奇迹。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >总结起来这个产品市场成功要素一是价格低，二是品牌好，三是质量稳定；技术成功要素是通用化，标准化，操作简单，外观美观。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在后面多年的产品设计中，几乎工厂所有的主要畅销产品都是我做的结构设计，或者是我带徒弟做的，实际上那时已有竞争，往往是几个设计小组同时做设计，然后由销售人员和厂长确定。产品的成功，也让我的技术日趋提高，也为我瀛得了很高的厂内外声誉，在我远没有被评为工程师的时候，从厂长开始就上上下下都叫我<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>“马工”。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >那时候的厂长是一个比较通达的人，一般我们完成一个产品设计就会让我们休整一下。由于多发奖金还是一个困难的事，所以一般都会安排骨干到各地去做市场调研，调研也是真的包括与经销商沟通和自己站柜台做一天营业员。然后会游览一下当地风景。除此之外我自己会利用这段时间和下班时间研究一些技术。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >80</span><span style=""   >年代家电产品大量的使用塑料零件，这就带来了材料选择，零件设计和零件连接的很多问题。我开始研究其中的塑料零件连接技术，国内出版物已远不能回答我思考的问题。从<span lang="EN-US"   >1985</span>年开始，我开始读《<span lang="EN-US"   >Plastic Design Forum</span>》，《<span lang="EN-US"   >Plastic Engineer</span>》，《<span lang="EN-US"   >Toshiba Review </span>》，<span lang="EN-US"   > IEEE</span>的大分子化学卷<span lang="EN-US"   >(</span>记得是第<span lang="EN-US"   >34</span>卷<span lang="EN-US"   >) </span>等杂志 以及《<span lang="EN-US"   >Electronic Packaging Handbook</span>》等年鉴类书。陆续跟踪和了解这些技术<span lang="EN-US"   >6</span>～<span lang="EN-US"   >7</span>年时间，仅其中的小小自攻螺钉连接技术就花了<span lang="EN-US"   > 2</span>年时间。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >当时自攻螺钉刚刚开始在国内用，我们的产品仅在一些不重要的连接采用自攻螺钉，原因是它容易滑牙。而打开<span lang="EN-US"   >**</span>的产品已<span lang="EN-US"   >100%</span>采用自攻螺钉。用自攻螺钉能大幅度降低装配成本，我觉得这就有研究价值。我跑了南京图书馆，江苏科技情报所，江苏电子科技情报所，南大图书馆，南京化工学院图书馆，上海图书馆，上海科技情报所，北京图书馆，电子部第四研究所，沈阳<span lang="EN-US"   >790</span>厂（电子部紧固件研究所）我收集了所有的国内外的相关书籍，杂志，中日美英德<span lang="EN-US"   >ISO</span>标准，约<span lang="EN-US"   > 200 </span>万字。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >1885</span><span style=""   >－<span lang="EN-US"   >1990</span>年在读这些英文资料时，我陆续将其翻译发表在省和国家级技术刊物上计<span lang="EN-US"   >4</span>万字以上。这给我带来了<span lang="EN-US"   >2</span>个好处，一个是我成了中国电子学会的会员，一个是在<span lang="EN-US"   >90</span>年我被评为工程师时免考英文。<span lang="EN-US"   >1992</span>年，经过多年的翻译，研究和实际设计经验积累，我发表了自己<span lang="EN-US"   >10</span>万字的论文集《塑料零件连接技术》（中国音响工业协会出版），这时我在金属零件，塑料零件的设计，连接技术，包括对这些材料的知识，以及冷冲压工艺，注塑工艺有了很多独到的经验，那时候我在工厂对技术问题已不敢随便说话，因为你一说，别人就会认为是对的，就会照着做。在行业已小有名气。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >写技术论文不仅使自己进一步理清思路，也深化对问题的理解，因为你写得时候才会发现，有些问题原来自己并不很清楚，给别人讲课也是如此。写和讲都是与己与人有益的事。我后来对工程师的要求都是要会做，会写和会讲。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >电子产品的结构设计，是一个比较容易上手的工作，有<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>年工作经验的大学生就能出图，但在选材，工艺性，易装拆性，可靠性，电磁兼容性方面的技术和经验真的需要多年积累。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我后来<span lang="EN-US"   >Interview</span>新员工时经常问什么是电子产品的设计？什么是结构？什么是结构设计？但得到的多是大学教材里的隔靴搔痒答案。我的体会是：“结构”对零件而言，就是形状和材料；更多的情况“结构”是对<span lang="EN-US"   >2</span>个或以上有装配关系的零件而言，就是<span lang="EN-US"   >2</span>个或以上的零件它们的结合处的配合形状；对整机“结构”<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>而言就是指零部件的布局。因此结构设计就是用最简的形状，合适的材料，精巧的连接，合理的布局实现功和性能的要求。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >10</span><span style=""   >年努力，我实现了做工程师的理想，<span lang="EN-US"   >1990</span>年我拿到了省里发的工程师证书。在国企的日子，受到的教育使自己一直感到自己是工厂的主人，是为自己在奋斗。虽然那只是一个空头支票。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >4. </span><span style=""   >男怕入错行 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >94</span><span style=""   >年前后，音响行业一方面由于香港大量的<span lang="EN-US"   >CKD</span>在南方地区组装，而香港的模具制造水平比内地高一个数量级，产品的外观漂亮；一方面由于江浙地区乡镇企业依靠“星期日工程师”也大量的生产音响。这些产品仅注重基本功能，摒弃了国企按国标追求的高性能指标（这些指标对一般用户确也没有用）。这两类公司的生产效率比较高，产品有价格优势。因此音响市场竞争日趋激烈，国企产品的技术优势渐失。而国企固有的顽疾确在内外因素作用下发作起来，日子开始难过。国家的统计数据表明，由于恶性竞争，全行业处于亏损状态。而那两类公司多数不在行业统计数据之内，维持在微利状况。在这个行业做了十多年，应该说心中是有数的。<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>“男怕入错行”在一个走向没落的行业里，可能你再投入多少力气，回报也是微薄的。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我开始学习计算机技术，我把我的一台<span lang="EN-US"   >486</span>工作用机拆了一遍又一遍，<span lang="EN-US"   >
DOS</span>系统也装了无数次，<span lang="EN-US"   >DOS</span>命令也用熟悉了，这时市工会职校的计算机中专班聘兼职老师，我就凭着一张<span lang="EN-US"   >DOS</span>激活盘，一张<span lang="EN-US"   >DM</span>盘，一张<span lang="EN-US"   >KV100</span>杀毒盘和我从计算机报刊上得到的知识，开始教计算机维修课，实际上当时的计算机全是进口兼容机，品质相当好，故障也确实基本上是软件操作不当造成，我记得稍为难一点的是在计算机升级的时候要会编辑<span lang="EN-US"   >Config.sys</span>和<span lang="EN-US"   > Autoexec.BAT</span>两个文件和主板上的倍频跳线。事实上我的课还很快吸引了很多南工和南大的本科生来听，我当时每周<span lang="EN-US"   >4</span>节课，每上两节课，要备<span lang="EN-US"   >8</span>小时的课，我订了当时所有的计算机报刊，当学生问的问题当时答不出，我一定会弄清楚，下次课说明，其实带着问题的学习是效率最高的，我也经常在珠江路计算机一条街上看别人怎样装计算机。我的性格中有敢闯的一面，也有喜欢探求新知识，打破沙锅问到底的自然习性。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >94</span><span style=""   >年我有了一个到“西电”去进修计算机技术的机会。一年以后，我看到报纸上同创登的招聘广告，就以普通工程师的身份，应聘同创集团新品研发中心结构设计工程师，面试当场即被录取，并让我即去上班，面试我的是新品研发中心总监徐一宁，这是一位做人很正的领导，只是由于有时表达过于直白，而没有更好的位置，我对他仍然心存感激，是他让我赶上了这一波计算机热的大潮，开始了<span lang="EN-US"   >10</span>年的<span lang="EN-US"   >IT</span>之旅。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >92</span><span style=""   >年有个同事要到深圳去闯天下，我们做的音响这一行，在深圳薪水高很多。因为他是做电路设计的，我是做结构的，所以希望我和他一起去，这样就可以搭档做产品。我对他说，你还没做好准备，技术上你还要积累，而且这个行业现在我也不看好。结果是他去了，我没去。一年后的年底，他神采飞扬的回来，请大家吃饭。说他月薪<span lang="EN-US"   >6</span>千块。而我们那时月薪才<span lang="EN-US"   >1</span>千块，结果是有一批人跟出去做音响。而我是准备转行了，再过<span lang="EN-US"   >2</span>年后我也到了深圳，只是做了<span lang="EN-US"   >IT</span>业。后来那些同事碰面，都说公司经营状况不好，薪水几年没加。而我的薪水确实是年年在加。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >这一次的人生转折，是一次正确的选择。我的大部分同事还在国企里面耗着，有时看到他们面带菜色，就知道他们的处境。成功的转折来自于对社会，行业，自身的了解和极积的准备，来自于忧患意识，这样才能抓住时机。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >5. </span><span style=""   >化蛹为蝶 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >长期在一个地方工作对外界真是不太了解，开始工作的一个月给了我很多刺激。首先是薪资，面试时，我要了我在工厂时同样的月薪。面试者即是我的主管，笑了笑，一口答应。结果到第一个月发工资时，我得到了<span lang="EN-US"   >2</span>倍的钱，其中一半是奖金。我悄悄问了一下同事，原来工作<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>年的大学生也跟我拿得差不多。我恍然大悟，怪不得面试时主管笑得暧昧。<span lang="EN-US"   >IT</span>行业的薪资真是跟传统行业不好比。这也大大刺激了我的工作热情。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >第二件事是不象我们电子行业设计师每人配备几十万圆的电子仪器，这里只是每人一台<span lang="EN-US"   >PC</span>，所谓硬件工程师做的只是做用户层面的测试，他们多是工作<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>年的大学生。而结构工程师倒是需要做真正的设计。只是在这里共事的同事，他们离真正的设计实在太远，虽然也能用<span lang="EN-US"   >AutoCAD</span>画零件图形，但对结构设计的要素掌握。这样的状况在后来的日子我多次碰到，包括我后来与台湾结构工程师的多次交流，我更是对自己有信心：我在技术上走得够远。顺便说一句，台湾称结构工程师为机构工程师是比较准确的，“机构”者“机械结构”也。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我认为中国的<span lang="EN-US"   >IT</span>企业之所以没有成功的掌握深层次技术与其<span lang="EN-US"   >R&amp;D</span>的组成人员来源有关，他们差不多都是学计算机的新大学生。很多公司靠这批没有工厂制程经验的新人做板级研发时，遭遇了失败而在研发路上缩了回去。当时有大批的成熟电子工程师，他们没有及时的向计算机行业转移，是国家的损失也是他们个人的损失。因为实际上除了软件工程师，其它象硬件，测试，结构，品质工程师，他们都需要长期的经验积累。如果有一批人转了过来，再加上新大学生计算机知识的优势，至少中国的<span lang="EN-US"   >PC</span>的板级研发有的跟台湾一拼。今天说这些，确实是过时，但在<span lang="EN-US"   >IT</span>目前又一轮的技术变革面前会有借鉴之处。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >第三件事是称谓，“嗨，你过来一下”，这是部门一个同事开始对我的称谓，从被上下尊称为“马工”到被称为“嗨”，也使我清醒的认识到：在一个新环境，过去的光环已不存在，你的能力，你的绩效，别人一概不知，再加上每家公司对一些人和事的处理各不相同，一定要在心态上做好调整，脑子要归“<span lang="EN-US"   >0</span>”，要用行动去证明自己，而不要对种种的新状况耿耿于怀。证明自己需要时间，不过这次时间来的很快。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我很快开始了一款<span lang="EN-US"   >PC</span>的结构设计，两个月的时间不仅完成了结构设计还用数控折弯机做了<span lang="EN-US"   >5</span>台样机，面板也用手板做成。在新品评审会上受到了公司总裁的好评。他问这是那个设计的，当知道是我时，他考了我个问题：“散热有几种方式”？我说：“对流，传导，辐射”，他又问我具体的散热设计方案，除了较常见的方案，我还说了“液体散热”。这是我第<span lang="EN-US"   >1</span>次与总裁对话。会后我成为研发中心结构设计部长，有<span lang="EN-US"   >10</span>个成员。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >人生的经历是财富。我工作的前几年是设计仪器结构，主要设计金属薄板零件，后来设计音响结构主要是设计塑料零件。现在设计<span lang="EN-US"   >PC</span>机箱，面板用塑料零件，机架用金属薄板，正好将我的技术积累全部用上了。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >一个月后我们一群工程师在南方选<span lang="EN-US"   >PC</span>机箱，我分析“保利得”一款机箱的软驱按键装配方式在运输中时可能会有问题，但有人说没有问题，台湾人设计的还会有问题？我们自己设计的才会有问题。结果上市后有<span lang="EN-US"   >30%</span>产品软驱按键因运输原因被撞断，我在“保利得”与他们的工程师商讨解决方法，他们不承认是设计问题，认为是我们的包装设计问题，我请示了老板后，跟他们说，先解决问题，我重设计一个按键，模具费我们公司出，你们帮忙找一个可靠的模具厂，加班做出来。对方同意，结果我花<span lang="EN-US"   >4</span>小时画完图纸，在<span lang="EN-US"   >2</span>倍模具费的刺激下，<span lang="EN-US"   >7</span>天后零件成功，<span lang="EN-US"   >1.5</span>万台已出货机器换了此按键后，问题全部解决。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >这时我们再来讨论责任，先是与“保利得”的业务经理谈，我一一把可能的原因排除后，最后只能是设计问题的分析说给他听，在我说的过程中，我每说一点，他就说“是”，一直到我说完还说“是”，让我觉得他同意了我的观点，却不料轮到他说时，他说“我不是做技术的，要找我们的工程师跟你们谈”。实际上他是懂技术的，这位在美国生活了<span lang="EN-US"   >10</span>年的左家骅经理让我懂得了职业立场和谈判技巧。后来“保利得”派来了据说已退休的总工程师和一个包装设计工程师，在事实面前，他们还是体现了固执。虽然他们比我们更会说表面上的“对不起”。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >半年后集团新品研发中心搬迁至深圳，总裁兼总监，我任常务副总监，我们原来的总监任集团总工程师。开始了我们一群人<span lang="EN-US"   >5</span>年的激情岁月。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >6. </span><span style=""   >激情岁月 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >那时候<span lang="EN-US"   >IT</span>公司头上有许多光环，在<span lang="EN-US"   >IT</span>公司工作是时尚，先进，高薪，有前途的象征，公司又是军方和省市政府投资的公司，总裁又极富鼓动力。公司在高速扩充中，尤其是销售人员，很多毕业一年的大学生，就做了负责一个省的销售公司经理。大家都热血沸腾，希望能成就一番事业。别说我们，就是国家科技部朱丽兰到南京时，开始不愿意到公司来参观，在市领导的劝说下，才答应来<span lang="EN-US"   >10</span>分钟，来了也不愿坐，想看一下新品陈列室就走，结果总裁才介绍了<span lang="EN-US"   >5</span>分钟，就把她吸引得坐了下来，一直听了<span lang="EN-US"   >40</span>多分钟。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我有了一个比较大的发挥空间，因地处深圳远离总部，也不太受干扰。总裁一周来交代一次工作。新品研发中心的工作地点是整一层楼，一半是大通间，一半是标准客房，有<span lang="EN-US"   >1600m2</span>。中心相对独立运行，为员工提供高标准的一日三餐和所有的生活日用品，也提供了对外有竞争力的薪水和劳保。员工没有生活之优。总裁的想法是希望封闭式的管理，按项目需求进人，做完了即可带着产品到工厂去生产。中心成立了项目管理部，硬件研发部，<span lang="EN-US"   >PCB layout</span>部，<span lang="EN-US"   >BIOS</span>研发部，测试部，结构设计部，整机研发部。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >研发中心相对独立实际上是一个有普遍意义的举措，因为不论是薪资待遇，工作模式，管理模式和研发人员特质都与公司其它部门人员不同，我看到有的公司在将窗户封闭的房子里做研发，大好新鲜空气被隔在窗外，进出要过安捡门。看到这些，更感到要好好交流各自经验和感受。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >新品研发中心的研发目标主要是<span lang="EN-US"   >PC</span>整机及主板和周边卡，我从南京挑了些大学生也在深圳招聘了一些工程师，整机研发主要是测试，结构设计和外观设计。主板研发开始想依靠台湾的主板研发工程师来带技术，也合作了半年，发现台湾工程师对自己的技术很保守，我们很难学到深层次的技术。台湾工程师撤回后，我们定的策略是依靠<span lang="EN-US"   >Intel VIA Chipset</span>厂商提供硬件技术支持，依靠<span lang="EN-US"   >AWORD</span>在<span lang="EN-US"   >BIOS</span>方面给我们支持。开始了我们真正的自主研发板卡。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我面临两种情况，一种是整机研发，其产品技术是我掌握之中的；另一种是主板和周边卡研发，其产品技术是我的弱项。这个问题是我碰到的新问题，实际上，它有一定的普遍性，象<span lang="EN-US"   >PDA</span>，<span lang="EN-US"   > Notebook</span>，这样的产品也是如此，可能没有一个人能掌握其需要的全部技术。对自己不深入掌握技术的产品研发，怎样管理，成为我后面<span lang="EN-US"   >10</span>年技术管理生涯的主要研究内容。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >为了确保成功，也为了满足大家跃跃欲试的心情，我们成立<span lang="EN-US"   >3</span>个主板设计组，设计同一块主板，我们根据能获得的技术支持资源，决定做<span lang="EN-US"   > VIA MVP4 Chipset </span>支持<span lang="EN-US"   >100</span>外频主板。同时，我们还安排了<span lang="EN-US"   >10/100MHz</span>自适应<span lang="EN-US"   >LAN card</span>和<span lang="EN-US"   >56K Modem card</span>的设计以及<span lang="EN-US"   >10/100MHz</span>自适应<span lang="EN-US"   >16port Hub</span>的设计，我们采用的主芯片都是芯片厂商刚刚推出的有市场竞争力的产品。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在研发中心副总的位上我自然比别人要多研究新产品的管理方法，而不是陷入技术细节之中。我制定了一系列的研发流程，包括一些子研发流程，例如，<span lang="EN-US"   > PCB layout</span>流程，测试流程；也建立了非常<span lang="EN-US"   >open</span>的技术研讨会制度；对每个产品都定了<span lang="EN-US"   >schedule</span>，每周<span lang="EN-US"   >update</span>一次，因为新手比较多，我特别注意了过程管理，在每一个研发阶段都会举行技术评审会。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >阶段性的技术评审会，对一个由成熟工程师组成的<span lang="EN-US"   >team</span>也许不重要，但对新手来说太重要了，我们是实质性评审，绝不是走过场。如我们规定设计者占项目奖的<span lang="EN-US"   > 80%</span>，主审者占项目奖<span lang="EN-US"   >20%</span>，责任同样是这样比率。在评审会上也纪录其它参与者的观点，特别是有不同意见时，不能达成共识时会记录在案，作为今后验证，绩效考核的依据。我还特别注意借助外力，象<span lang="EN-US"   >PCB</span>图会请<span lang="EN-US"   >VIA</span>的资深工程师帮忙<span lang="EN-US"   >check
</span>后再做。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >项目<span lang="EN-US"   > Schedule</span>也分解的很细，具体到什么人，做什么事，什么时间做，做得怎么样以及结果，<span lang="EN-US"   >10</span>年前我还没有看项目管理的书，完全是靠自己的琢磨。我的管理风格是盯的紧，能今天做的就不明天做。我也比较善于安排并行作业。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >项目管理主要是跟踪<span lang="EN-US"   >Schedule</span>的执行情况，协调各专业工程师的工作，负责供货商联系，安排做<span lang="EN-US"   >Sample</span>，做产品会议的招集人，总的来说就是做设计以外的所有工作，让工程师专心关注技术。而产品的规格一般都是项目管理， 研发主管或更高领导也要参加讨论的事。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >各研发部长（象硬件研发部长）与项目管理怎么将职责划分清楚？我的做法是在工程师没有项目时，研发部长负责日常管理和技术培训，在工程师有项目时，负责审核工程师的工作输出，并做技术指导。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在那段日子里，大家是鼓足干劲做事，<span lang="EN-US"   >MVP4</span>主板的<span lang="EN-US"   >Chipset
Datasheet</span>分章研读，再交互讲解，<span lang="EN-US"   >schematic</span>也经讨论确定，在<span lang="EN-US"   > PCB layout </span>时我们走了弯路，我们用的软件是<span lang="EN-US"   >Power PCB</span>，<span lang="EN-US"   >3</span>人一组，每人做<span lang="EN-US"   >8</span>小时，<span lang="EN-US"   >24</span>小时连续作业，结果因没有一个水准高的做统筹，各人思路有差距，反复较多。在几年后，我们用<span lang="EN-US"   >Allegro</span>软件，加上一个有经验的工程师做统筹规划和任务分配，可以做到<span lang="EN-US"   >9</span>人布一块板，一周完成（<span lang="EN-US"   >3</span>组分别负责布一部分，<span lang="EN-US"   >3</span>人一组，每人做<span lang="EN-US"   >8</span>小时，<span lang="EN-US"   >24</span>小时连续作业，<span lang="EN-US"   > Allegro</span>有自动拼图功能）。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >经过<span lang="EN-US"   >4</span>～<span lang="EN-US"   >5</span>个月的努力，这批新产品在北京新世纪饭店开了新产品技术发布会。作为业界的一匹黑马，又推出了这样一些自己设计的产品，业界是将信将疑。后来有一批记者和官员来参观了我们的研发中心才冰释前疑。这些产品都投入了量产，在<span lang="EN-US"   >MVP4</span>主板的量产中，代工厂只能做到<span lang="EN-US"   >95%</span>的良率，后来分析是设计余量不够，具体有两个因素，一个是元器件的“温飘”，一个是元器件制造的本身偏差，这为我们提供了一个教训。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >一个好的设计师，他一定要用心做事，技术上的事是躲不过去的，今天没有仔细去核算上下两种工作极限的状况，明天一定会让你十倍的头痛。“天道酬勤”就是讲的这个道理。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >我当时对人的要求是要主动，细心，开放，诚实。对所有进入中心工作的成员，都宣讲清楚，每一条都有正反方面的事例说明。如果违背，就是否决项，不能在中心工作。所以大家都非常<span lang="EN-US"   >open</span>，技术上毫无保留。用我们总裁的话说就是：“大家都是短板子，但很多短板子也能围成大木盆，能盛很多水”。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我平时管理风格还是很认真的，对重复出现的错误我会严肃指出，包括后来我选的副手。开始有的人是很不适应的，但后来看到是一视同仁，而且确实是为了把事做好，也慢慢能适应了。我是一个急性格和追求完美的人，因此也是一个对自己和别人要求都比较高的人。我有我的一些不合常理的逻辑：当两人同出错，我会批评跟我比较近的人；发产品奖金有争议时跟我比较近的人，我也会给他少些。那时大家很融洽，每周末我基本上都会轮流请部分员工吃饭，这也是一种人际关系平衡方法。我从不报销，虽然我有那个权力。不报销一是省得解释，二是自己很自信，从国企出来，原来外面的世界很精彩，自己很有竞争力，相信“千金散尽还复来”。<span lang="EN-US"   ><span style=""   >&nbsp; </span></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我的性格本质上是属于急性格的一类人，但一般时候看不出来。因为我在南京时，公司午餐时间，我经常会讲一些故事，我讲的故事往往是我自己独特的思考，而不是说书本上的东西，这就引起了同事们的兴趣和讨论，常常是到了下午上班时间一群人还围在哪里，津津有味的说不完。比如我说过，不要看现在江南这么富，其实原来江北更发达，理由之一就是：不论吃什么，江南人一概用“吃”一个字，而江北人则会用很多不同的动词来说，象“喝”酒，“嗑”瓜子，“叨”菜，由此我推论中国的文明起源于黄河流域。我还说过，为什么重庆出美女，是因为抗战时期中国精英大批去重庆的原故，其它象米脂，扬州，上海，哈尔滨特征都是历史上都有大批外来人口的迁徙。所以那时我给人的印象是随和的人。在深圳容易发脾气，客观的说，主要是求胜心切，压力大。我们总裁是郭台铭那一类型的人。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >我以前单位的党委书记，见到所有员工都会微笑，后来做了厂长，压力大了，很快脸上没有了笑容，一年后又做了党委书记，笑容才又回到了他脸上。我的好几位同事，几年后，在做了部门经理后，都对我说过，现在可以理解你那时为什么那么急。不过我必须承认的是，直到今天我的修养远远的不够，我仍然做不到“喜怒不露于色”。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >公司是较早注重企业文化的企业，每个员工进公司都会发企业文化手册，其中的“同心创业，共享未来”，“陀螺理论”给我影响很大，前者是有现实的利益，公司买了一大块地在为员工盖宿舍，是看得见的；后者是很形象，说得是：陀螺在旋转时才不会倒下，而且越快越稳定，越不会倒下，所以大家要拼命工作才能赢。企业文化是一个想永续经营的企业的基石。其实好的企业文化道理不难理解，难的是员工的真正认同和将其溶于工作中。我们那时的状况，一方面公司急于要看到成果，一方面在深圳的年轻人一般都是干一份活，拿一份钱。因此能够凝聚一批人材，在人员流动率比较高的深圳保持了人员稳定，并且快速出了成果和人才。今天回头看，主要靠的是<span lang="EN-US"   >3</span>点：第一是公司有一定知名度，薪资福利对外有竞争力；第二是大家有事做，有技术学；第三部门风气正，奖罚分明。第一条是公司作用，后面的两条有我和大家的努力。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >7. </span><span style=""   >走入海尔 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在一个社会大变革的时代，有时不是你不想稳定，是企业做着做着就不行了，你不得不做新的选择。<span lang="EN-US"   >1998</span>年国家规定军队不准经商，在军队撤资，集团人事变更，经营管理等各方面的影响下，公司从全国性的大公司收缩成地方性的小公司，研发中心难以为续，宣布解散。我面临新的选择。这时国内有很多家电企业想进入<span lang="EN-US"   ><span style=""   >&nbsp; </span>IT</span>业。经过几年的<span lang="EN-US"   >IT</span>产品研发，我对<span lang="EN-US"   >IT</span>产品有了一定的技术和市场感觉。我给<span lang="EN-US"   >TCL</span>李东生，海尔张瑞敏，康佳陈伟荣，新科秦志尚，创惟黄宏生分别写了<span lang="EN-US"   ><span style=""   >&nbsp; </span>mail </span>和<span lang="EN-US"   >Fax</span>。海尔回复最快，结果是在深圳成立了深圳海尔信息科技有限公司，我们重新招聘了员工，这实际上是一个<span lang="EN-US"   >Design House </span>有<span lang="EN-US"   > 25</span>人。 海尔集团总裁杨绵绵兼总经理，我为常务副总经理，我在同创的同事杨宏春为副总经理。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们研发了“海尔小超人”<span lang="EN-US"   >PC</span>，它实现了语音识别控制计算机，当你进行语音训练后，它就可以根据你的指令，自动开关机，并可以用语音提示你已安排的日程，语音识别部分做在一个兔子形状的体内，兔子手臂会动作，眼睛会发光。通过<span lang="EN-US"   >USB</span>接口与<span lang="EN-US"   >PC</span>相连。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们还研发了基于<span lang="EN-US"   >ARM</span>的以<span lang="EN-US"   >Creator 7209</span>为主芯片的<span lang="EN-US"   >MP3</span>，其功能包括播放多种格式的音乐如<span lang="EN-US"   >MP3/WMA/AAC</span>，录音，文件存储和管理，复读机的所有功能。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >这两个产品研发时有两种意见，一种意见是先做最简功能推上市，然后再进一步做成系列产品，即一次规划，分次实现；一种意见是，一次规划，一次到位，能够一鸣惊人。我的想法是前一种。杨宏春和集团的计算机公司技术主管的想法是后一种，鉴于计算机公司实际上是我们的客户，也考虑海尔付得起这个学费，我同意按他们的想法做。结果由于技术上要解决的问题太多，产品上市一再推迟。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >工程师研发产品往往追求性能指针，设计的产品能代表什么什么水平，追求他认为有用的多而全的功能，往往忽视市场的时间需求和用户的实际需求。有时候，当影响公司的大局时，这种情况是一定要扭转的。如果不影响公司的大局，通过事实来教育新人，可能效果更好。因为你不让他试，就证明不了利弊。付这种成长的代价，其实也是体现一种企业文化。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >海尔有个“<span lang="EN-US"   >8</span>号会”，即每月的<span lang="EN-US"   >8</span>号，集团的经理级以上的干部会，大家天南海北的集中到青岛，地点在总部的海尔大学的阶梯形教室，开会时用多个投影仪，每人坐位上有麦克风。一般会上有三个内容：第一部分是各生产公司，销售公司的营业额排名，有准备好的报表，时间<span lang="EN-US"   >10</span>几分钟；第二部分是重点，是案例讨论，案例有专人准备。用录象，录音客观反映这一个月发生在某个生产，技术，销售环节的一件事。然后讨论这件事的全过程，哪个环节做的对，哪个环节做的不对，对在那里，错在那里，怎样改善，这个案例重点放在管理层面，避开技术细节，以让大家均可参与。讨论由张瑞敏自己主持，这个时候，特别能体现张瑞敏的视角独到之处，体现他的逻辑能力，洞察力和把坏事变为好事的能力。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >海尔还有个“<span lang="EN-US"   >9</span>号会”，即每两个月的<span lang="EN-US"   >9</span>号开新产品推介会，由在全国各地的<span lang="EN-US"   >10</span>个研发公司和集团空调，冰箱，电视等公司的设计所介绍这两个月来的新产品，场面做得很大，有实物，有图片，有演示。特别要介绍产品有什么市场“卖点”，产品处于同行的什么水平。然后由张瑞敏点评，最后由各销售公司经理填销售需求表，任一产品销售需求不足<span lang="EN-US"   >3</span>万台，不能投产。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >1999</span><span style=""   >年，深圳高交会期间，我陪他看了展会，并请他到深圳海尔科技给工程师们讲了话，张瑞敏说，做事关键是思路，海尔有品牌优势，服务优势，量大优势，你们做技术要善用这些优势，要借力去做技术整合，否则这几个人能做成几件事。他认为，核心技术固然是核心竞争力，核心技术还是为了赢得用户，在核心技术一时难求的时候，我们如果已经赢得了用户和用户的忠诚度，这就是核心竞争力。张瑞敏内心充满做第一的激情，但又充满理智。外表气定神闲，说话从不急躁，但一语中的，极富人格魅力，他是中国少有的思想家型企业家。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >海尔在全国各地网罗人材，在北京，上海，深圳，西安等地成立了<span lang="EN-US"   >10</span>个研发公司，以改变海尔的技术形象，这些研发公司投入较高，人员待遇也比集团本部的技术<span lang="EN-US"   >/</span>管理人员要好，在中下层也带来一些管理上的冲突。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span>8. </span><span style=""   >学历 能力 绩效 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >没有读一所名牌大学是我心中永远的痛，这种痛主要来自于每一次的工作转变，倒不见得再读大学可以多学多少知识。像海尔的各地研发公司经理除了我其它都是博士。其实后来工作中读书还是有机会的，在熊猫集团东方厂的时候就有过一次机会，后来因为手上有设计项目，没有去成，后来只去西电有过一次短期的培训。到了同创后，有了读工程硕士的机会，我报了名，刚刚开始读一个月，总裁要我去深圳做研发，我说要在南京读书，他说，你看硕士毕业才多少工资？你去深圳都超过了那个工资，还要那个虚名干什么？结果是我又一次失去了再读大学的机会，真是“失之东隅，得之榆桑”。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >在今天的职场，很多的国内企业第一看重的还是文凭。而外企的做法是对<span lang="EN-US"   >5</span>年工作经验以内的人，其文凭占比较重要的录取分数；对<span lang="EN-US"   >5</span>年工作经验以上的人主要看其能力，绩效，越是工作年龄长，其文凭占的录取分数就越小。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >这里我抄一个小故事<span lang="EN-US"   >: </span></span></p>

<p><span style=""   lang="EN-US"   >********************************************* </span></p>

<p><span style=""   >学历代表过去，只有学习力才能代表将来 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >有一个博士分配到一家研究所上班，成为学历最高的一个人。有一天，他到单位后面的小池塘去钓鱼，正好正副所长在他的一左一右，也在钓鱼。他只是微微点了点头，心想：跟这两个大学毕业生，有啥好聊的呢？不一会儿，正所长放下钓竿，伸伸懒腰，蹭蹭蹭从水面上如飞地走到对面上厕所。博士眼睛睁得都快掉下来了。水上飘？不会吧？这可是一个池塘啊。正所长上完厕所，回程同样也是蹭蹭蹭地从水上飘回来了。怎么回事？博士又不好去问，心想自己是博士生哪！过一阵，副所长也站起来，走了几步，蹭蹭蹭地飘过水面上厕所。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >这下子博士更是差点昏倒：不会吧，到了一个江湖高手集中的地方？想着， 想着， 博士生也内急了。这个池塘两边有围墙，要到对面厕所非得绕十分钟的路，而回单位上又太远，怎么办？博士生也不愿意去问两位所长，憋了半天后，也起身往水中跨：我就不信大学毕业生能过的水面，我博士生不能过。只听到
“噗咚”<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>的一声，博士生栽到了水里面去啦。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >两位所长赶忙将他拉了出来，问他为什么要下水？！博士反问：「为什么你们可以走过去呢？」两所长相视一笑：「这池塘里有两排木桩子，由于这两天下雨涨水正好在水面下。我们都知道这木桩的位置，所以可以踩着桩子过去。你怎么不问一声呢？」
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >学历代表过去，只有学习力才能代表将来。尊重经验的人，才能少走弯路。一个好的团队，也应该是学习型的团队。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >********************************************* </span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >一般的说，本科生比大专生基础要好，工作能力也强些。但硕士就不一定比本科生工作能力强，博士也不一定比硕士工作能力强。尤其是长期来看，很难看出规律。所以说学历好，不一定能力好，能力好不一定绩效好。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >能力好，不肯出力，自然绩效不好；还有一种情况是个性问题，我就碰到过一个工程师在接研发项目时会提种种条件，往往因此失去做项目机会；难得接到项目，研发时碰到问题又老是说别人的问题，搞得左右都不协作，绩效当然不好。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >世上有两种成功的工程师，一种人是靠过人的聪明<span lang="EN-US"   >.</span>。在熊猫集团东方厂时我的上司丁石藤（现在复旦大学工作），后来在<span lang="EN-US"   >ABIT</span>时的<span lang="EN-US"   >YT</span>－<span lang="EN-US"   >lee</span>都是属于高智商的人，他们读书不吃力，成绩优秀；工作绩效也好，也不用靠加班熬夜。但这毕竟是属于很难找到的极少数人，我这么多年和那么多工程师共事，真正聪明的人也就碰到<span lang="EN-US"   >2</span>～<span lang="EN-US"   >3</span>人。大部分人取得成功，确实要靠方法和勤奋。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp; </span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >9. </span><span style=""   >人脉关系 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >工作中建立起来的人脉关系因该说是自然形成的，互相间觉得比较投缘，就多了一些来往，各自工作中碰到技术问题，也可以交流一下。因为工作了<span lang="EN-US"   >20</span>年，走过几家公司，各种知识背景的人都认识一些。每一次的工作变更，我都会把工作交代的清清楚楚，好聚好散，也使我有比较多的人脉关系，这给后来的工作带来很多方便。也有些人并没共过事，仅一面之交，但也为今后留下了某种机缘。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我多年面试新员工有一个习惯，就是帮面试者倒一杯水。这源于自己的一种感觉，人家大老远赶过来，心情又紧张，喝口水应该是需要的。还有就是第一天上班的新同事一般不会带自己的杯子来，我也会让助理给他倒一杯水。而且在从面试到新同事第一天上班，整个过程，我在公司<span lang="EN-US"   >HR</span>规定的框架内，要求部门助理做到让新同事感到“宾至如归”。来公司谈事的供货商<span lang="EN-US"   >FAE</span>或<span lang="EN-US"   >sales</span>到午饭时，我都会对同事说，多定个盒饭，让人家吃过饭再走。这几件小事使我添了一些朋友，也为公司赢得了一些口碑。有的没被录取的人给我写<span lang="EN-US"   >mail</span>，说应聘多少公司，仅在你们这里受到了礼遇，以后有机会还是想来工作。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >“以人为本”被很多公司列入企业文化，但怎样变为员工的行动准则，落实在细小的环节，那真是件困难的事。面对弱者，怀抱怎样的心态，是一种修炼。我一直有一个做人的原则就是：遇弱不强，遇强不弱。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >当然问题的另一面就是要交接比自己强的人，这就象下棋一样，要找高手，你才容易进步，而且能帮你的，往往是比你强的人。弱者和强者之间怎么能取得一种资源互补，要运用智能。这个技术问题在后面的“技术传承”中再介绍。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >社会变革和技术革新都会让一些企业掉队，经历了潮起潮落，心中的忧患意识是很强的。所以我一直以来都非常关注行业的技术发展趋势，也特别关注行业前几名企业的状况，也有意识的参加一些技术论坛，展会认识一些人。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >说起来<span lang="EN-US"   >1980</span>年代，各行业的技术人员都有全国性的技术交流会，对技术人员的水平提高和技术的发展是有益的；现在各公司技术相对封闭过分了，说起来是为了公司各自的利益，但得不偿失。好在有网络，在一些专业网站上，还是能交流很多知识的。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >10. </span><span style=""   >台资企业 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >上个世纪末，大型台资<span lang="EN-US"   >IT</span>企业纷纷投资长三角地区，特别是在苏州地区。台湾主板厂<span lang="EN-US"   >ABIT</span>公司也又一次布局大陆。他们找到了我，希望我去帮<span lang="EN-US"   >ABIT</span>筹建和管理<span lang="EN-US"   >ABIT</span>大陆研发中心，这缘于前几年在技术业务来往中建立的人脉关系和相互的了解。我想有几件事给他们留了印象：有一次是他们供给我公司的产品有品质问题，我一层层追踪得不到结果的时候，打了很多电话，得到了他们研发处长的家庭电话，打过去，终于联系上了，得到了明确的答案。那天是中秋节。我不是不知道公事一般不打家庭电话，但是做事突破常规，才能做成别人做不成的事。还有一次是一个项目的论证会，当报告者在报效益评估数据卡壳时，我报出一连串数据，帮助解了围。另外是他们看到我们的团队人员工作时的活力和我们的工程师与他们的工程师有冲突时我的客观处理态度。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >鉴于我家在南京和他们给我的良好待遇，<span lang="EN-US"   >2001</span>年初我接收了这个中心常务副总经理的职务<span lang="EN-US"   >(</span>无总经理<span lang="EN-US"   >)</span>，从“<span lang="EN-US"   >0</span>”开始筹建。我将深圳海尔信息科技有限公司常务副总经理的职务向海尔集团推荐，交给了我的副手杨宏春接任。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >ABIT</span><span style=""   >股票当时刚在台湾股市上市<span lang="EN-US"   >2</span>年，成立这个研发中心可能更重要的是要一种效应，因为这样，台湾股票就会涨。台湾的总经理一再强调不要急，技术人材的培养要很多年，招聘应届毕业生慢慢培养。要有“永续经营”的思想和做法。这也给了我一个比较好的环境，我那时已给自己定位为技术管理，因为要做大量的各种培训，也使得我又一次去<span lang="EN-US"   >update</span>和总结一些研发流程和绩效考核方法，也去读了大量的管理类书。这里面有很多话题，放在后面介绍现在还是先说台资企业。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >大型台资<span lang="EN-US"   >IT</span>企业在台湾都是股市上市公司，他们一般都已经过了资本的原始积累阶段，经营上比较平稳，不是那么急功近利，特别是在遵守政府法规，对员工的薪资福利方面是“明码标价”、很规范的，包括对新来的员工和流失的员工，做的够人性化。不像在大陆，即使有些名气的一些公司做的也很不规范。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >同规模的台资企业比大陆企业的管理要先进<span lang="EN-US"   >5</span>～<span lang="EN-US"   >10</span>年。就研发来说，<span lang="EN-US"   >EVT</span>、<span lang="EN-US"   > DVT review</span>，零件承认流程，新品试生产流程，网络化的<span lang="EN-US"   > Debug review</span>系统，<span lang="EN-US"   >ERP</span>中的料号系统，<span lang="EN-US"   >BOM</span>系统都是有效的流程化管理。就工厂的管理来说，均实现了基于网络的电子化管理，我所在<span lang="EN-US"   >ABIT
</span>的工厂——“罗礼科技”就有<span lang="EN-US"   >4</span>套系统：<span lang="EN-US"   >A.</span>用来考勤<span lang="EN-US"   > /</span>就餐<span lang="EN-US"   > /</span>门禁的刷卡系统；<span lang="EN-US"   >B.ERP</span>中的<span lang="EN-US"   >BOM</span>系统，<span lang="EN-US"   >MRP</span>系统，<span lang="EN-US"   >C. SFIS
(shop floor integrated system)</span>：现场信息整合系统中的品质管理系统；<span lang="EN-US"   >D.</span>文件管理和签核系统。这些管理手段大大减少了人为的失误。就整体上说，目前大陆和台湾的研发管理相比前者还是粗放形的，同样写一个流程，前者可能<span lang="EN-US"   >1</span>页就写完了，后者可能要写<span lang="EN-US"   >10</span>页。即便是<span lang="EN-US"   >IT</span>行业的领军团队也是有相当差距，而后者的精细，严谨，制度化已是普遍达到的水平。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >台资企业为大陆企业带来了技术和精细化的管理也提供了大量的就业的机会。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >台资企业薪资定的比较低是一个普遍现象比如同等工作，技术人员月薪在主要是欧美企业的苏州工业园区比主要是台资企业的苏州新区高<span lang="EN-US"   >500</span>圆。上海又比苏州工业园区要高<span lang="EN-US"   >500</span>～<span lang="EN-US"   >1000</span>圆。站在员工角度，看到苏州的房价飞涨，做<span lang="EN-US"   >10</span>年也买不起一套房子，生活水准不比上海低，如果没有特别的吸引，流动在所难免。上海贵的是房价，他们会选择在上海挣几年钱，再到一个合适生存的城市。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span><span style=""   >台资企业到大陆，短期主要是为了低价劳动力和降低其它营运成本，中长期是为了占领大陆的市场和在大陆的生存发展。因此台资企业对大陆技术人员是低技术的定位，他们并不想让大陆技术人员真正掌握高端技术。台资工厂纷纷迁移大陆，造成岛内工厂大批中低阶员工的失业对台湾研发工程师刺激很大的，这几乎牵扯到台湾家家户户，控制高技术向大陆转移也是台湾政府的政策。因此在台资企业学不到深层次技术也是正常的了，而且一旦发现大陆工程师的快速成长，出于自身利益的保护，也必然要利用机会、采取一些手段来遏制这种势头。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >台资企业在大陆的管理干部换人是频繁的，我了解的一些公司部门主管基本上一年换一个，并且还有同时派了<span lang="EN-US"   >2</span>个主管来，让下面的工程师无所适从，还有一些台干依靠权力在做管理，这都给工程师都留下了不好的感觉。在人际关系方面，台资企业有与国企一样的弱点。我在<span lang="EN-US"   >ABIT</span>苏州研发中心常务副总经理位上一做<span lang="EN-US"   >4</span>年多，而且还兼职<span lang="EN-US"   >ABIT</span>的工厂——“罗礼科技”的工程部协理，是个例外。这不仅是指时间，还指职务，在台资企业大陆的干部，一般做到课长就已经碰到了<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>“天花板”，这实际上也是大陆优秀人材把台资企业作为跳板的原因。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >台资企业到大陆，公司高阶干部清楚的认识到这是企业长期发展的必需动作，但中层和以下干部多数没有做好这个准备。他们还是没有做长期的打算，也不和大陆员工做比较深入的交流，他们的交际圈子还是台湾人，并没有想融入当地社会。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp; </span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >11.</span><span style=""   >建立规则 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >4</span><span style=""   >年前，我在读了大量的管理类书后，我得出了一个结论：那就是现代的管理源于西方，被<span lang="EN-US"   >***</span>发扬光大，再被台湾企业<span lang="EN-US"   >Copy</span>，再逐步从东西方多个渠道传到中国大陆企业。我写了有<span lang="EN-US"   >3</span>万字的文章发表在中心办的杂志上。不过今天我有新的想法，现代的管理思想有很多源于中国，实际上很多先进的理念都有老子的“道德经”孙子的“孙子兵法”和中国传统文化的影子。例如老子的“无为而治”，与<span lang="EN-US"   >GE</span>的“学习型组织”；孔子的“三人行必有我师”和<span lang="EN-US"   >GE</span>的“无边界管理”；百家姓中的“人之初，性本善”和
麦克雷格<span lang="EN-US"   >(Douglas McGregor)</span>的“<span lang="EN-US"   >Y </span>理论假设”；德鲁克<span lang="EN-US"   >(Peter F Drucker) </span>说的<span lang="EN-US"   >?</span>“发挥每一个人的长处设法使其短处不表现出来，因为人总是有弱点而且几乎不可能改变”几乎与中国的“扬长避短”如出一辙。等等。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在总部并没有给我一些企业文化资料的情况下，我从已往的工作经历中感到了它的重要性，我自己写了中心企业文化的文本，主要选取了同创，海尔，<span lang="EN-US"   >GE</span>，<span lang="EN-US"   >HP</span>，北电网络的一些理念。为了大家能够接受，又选了一些漫画家郑辛瑶的哲理漫画做辅助，还选了早期由张蔚，沉冰主持的<span lang="EN-US"   >CCTV</span>“对话”节目定期放给大家看<span lang="EN-US"   >(</span>买的<span lang="EN-US"   >VCD)</span>。用这些做为大家的日常行为准则和培训素材。针对每一条款，同时也做了案例说明什么是我们提倡的，什么是我们不提倡的，什么是我们不能触犯的<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>“天条”。我的体会是一开始就把“游戏准则”说清楚，这
样以后容易执行。实际上我在一一面试时会请面试者填一个“企业文化” 选择表，看看每一个人的选择，是否大致符合我们的要求，并且我根据他<span lang="EN-US"   >(</span>她<span lang="EN-US"   >)</span>的答案，会再交流一下各自的看法，对他说清楚今后中心的运作模式。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们很难说有些行为准则一定是对的，事实上行为准则是根据公司的状况的一种选择和正确的运用。是公司利益和员工利益的一种平衡。我始终认为，没有最好的管理只有最合适的管理，在<span lang="EN-US"   >A</span>公司行之有效的管理方法不一定在<span lang="EN-US"   >B</span>公司有效，因为背景条件不同。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >比如采用“团队合作”或“个人英雄”的做法。面对一个技术难题，在基本上都是新手的<span lang="EN-US"   > Team</span>中，我们一定会多用“团队合作”，先大家讨论分析再<span lang="EN-US"   >Debug</span>的做法，依靠大家的智能来解决问题。而对一个成熟工程师，我们多会用“个人英雄”的做法，因为可能对他来说，你还没讨论完，他已经搞定了，在这一阶段靠的是个人的力量。而后续，我们还是“团队合作”的思路，会要求他写出<span lang="EN-US"   >Debug</span>的思路和方法，以做技术传承。有的工程师不愿意写出来，这就是我们的否决项。不愿意做技术传承的工程师，不能在这个中心工作，这就是我们的一个“天条”。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >在建立规则方面，行为准则是一个最基础的东西。没有这些做共识，在做项目时会有很多磨擦产生，那个时候，一方面要解决技术问题，一方面再要讲基本观念就难了。因此在新手的培训期，要花大力气灌输。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >第<span lang="EN-US"   >2</span>层面的规则是组织架构，职责划分。中心的主要项目是主板和<span lang="EN-US"   >VGA</span>卡的研发，我们成立了项目管理部，硬件研发部，<span lang="EN-US"   >PCB layout</span>部，<span lang="EN-US"   >BIOS</span>研发部，测试部，人员规划<span lang="EN-US"   >50</span>人。我的打算是能够找到一些做为每个部的技术骨干，其它成员按公司要求招聘应届生。我也特别找了硬件方面很有功力的李兴中来做我的助手，<span lang="EN-US"   >BIOS</span>工程师一直没有合适人选。好在李兴中是软硬兼通，可以代管<span lang="EN-US"   >BIOS team</span>，鉴于李兴中还<span lang="EN-US"   >30</span>不到，不想管行政事务，只愿意处理技术问题，我们定下了项目管理部除了做<span lang="EN-US"   >PM</span>的工作外，还兼管中心的各部日常管理和考核的职责。我写了各部的职责说明书，这样就因地制宜的形成了矩阵式管理架构。我的体会是职责划分的越清楚越利于项目组成员的执行，做事效率也越高，不会出现一件事多人重复做，或者一件事没有人做。有一些公司不愿意在职责划分上花些时间将其定义清楚，而是一味地要员工“主动”，那不是一个好的做法。“主动”仅应该体现在两人职责的交接
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >处，没法写清楚的部分。“做我该做，说我该说”才是一个有序的团队。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >第<span lang="EN-US"   >3</span>层面的规则是研发流程，技术方法。我们又一次的<span lang="EN-US"   >update</span>板卡的研发流程，这一次我们在研发流程中特别加了几个内容，一个是每一环节的工作输入和工作输出；另一个是每一环节的所需标准时间，这个时间是按一个成熟工程师做的时候需花的时间。当项目有差异时，在绩效考核时再评估；再一个是加了每一个子项目的责任人栏目。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >技术方法是指<span lang="EN-US"   >PCB layout </span>规则，测试方法，<span lang="EN-US"   >Debug</span>工具<span lang="EN-US"   >/</span>仪器使用方法，在写这些文档时，我反复强调了一种观念：这些文档是写给新手看的，要尽可能的写的逻辑清晰，深入浅出。让水平低的人也能看得懂，才是高水平。我还特别找了一些中科院的院士写的科普读物来给我们的工程师做参考。在写每一个测试方法时，我们规定先写需要的测试部件和被测物，再写每一个测试要点的操作动作；再写操作动作后的屏幕反映<span lang="EN-US"   >(</span>剪贴屏幕<span lang="EN-US"   >)</span>，再用斜体写下测试注意事项。这样一份测试方法，一个应届生看了就能做个<span lang="EN-US"   >6~7</span>成，剩下的再问有经验的工程师就方便多了。这一方面相关的话题在后面的“技术传承”一节中还会介绍。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >第<span lang="EN-US"   >4</span>层面的规则是绩效管理。日常绩效考核主要是考核每个人的团队协作和部门公益性事务的执行情况。技术方面我们对项目的每个专业及每个阶段都规定了工作时间和工作品质的考核标准。例如，对主板的<span lang="EN-US"   >EVT</span>阶段<span lang="EN-US"   >(</span>设计后第<span lang="EN-US"   >1</span>次<span lang="EN-US"   >)</span>做的<span lang="EN-US"   >Sample</span>，工作品质的考核标准是一开机就能点亮，功能全部实现，主要性能指针也均合格为优秀；一开机就能点亮，功能全部实现，主要性能指针经过<span lang="EN-US"   >3</span>天以内的<span lang="EN-US"   >Debug</span>可以实现为良好；开机点不亮，功能，主要性能经过<span lang="EN-US"   >5</span>天以内的<span lang="EN-US"   ><span style=""   >&nbsp;
</span>Debug</span>可以全部实现为及格。<span lang="EN-US"   >(</span>制程问题除外<span lang="EN-US"   >)</span>。对主板的<span lang="EN-US"   >DVT</span>阶段<span lang="EN-US"   >(</span>设计后第<span lang="EN-US"   >2</span>次<span lang="EN-US"   >)</span>做的<span lang="EN-US"   >Sample</span>，则除<span lang="EN-US"   >BIOS</span>和<span lang="EN-US"   >Driver</span>以外，所有硬件问题特别是<span lang="EN-US"   >PCBlayout</span>问题都已解决为优秀；经过<span lang="EN-US"   >3</span>天以内的<span lang="EN-US"   >Debug</span>可以解决为良好；<span lang="EN-US"   >7</span>天内的<span lang="EN-US"   >Debug</span>可以解决为及格<span lang="EN-US"   >(</span>制程问题除外<span lang="EN-US"   >)</span>。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们会半年更新一次这些绩效考核方法，对这些内容，开始工程师们不以为然，但慢慢地大家感到了一种方便，一种公平，发现也是一种了解自己水平提升自己的方法。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >12.</span><span style=""   >管理方法 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >管理有<span lang="EN-US"   >4</span>个境界，最高境界是“无为而治”用现在的语言就是建立“学习型组织”，到达这个境界的团队已经高度成熟，会自我调节以适应外部的变化达成目标。第<span lang="EN-US"   > 2</span>境界是用电子和网络的手段，制定一系列流程，规则，方法让员工在既定的轨道里运行，使得团队不会出大问题。第<span lang="EN-US"   >3</span>境界是仅有粗糙的，不俱备可执行细节的规章制度，执行起来需要个人比较大的弹性发挥才能做事或经常需要请示上司才能做事。第<span lang="EN-US"   >4</span>境界是仅有的一些规章制度也被束之高阁，老板发号施令，公司员工基本上看老板脸色行事。这种公司中阶层人员善于揣摩上司的心态，适时调整数据，弹性解释事实；基层员工大多心存“你说你的我做我的”弹性作业。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我根据公司的状况，希望在研发中心能做到第<span lang="EN-US"   >2</span>境界。我比较推崇的管理方法是职责明晰，流程清楚，方法规范，公平竞争。从管理风格上我喜欢直面事实，不绕弯地说出自己的观点，尤其是对技术问题。但是这种管理风格我发现效果不好，其负面作用要存留很长时间、直到别人真正了解你才能消除。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >管理靠流程，规则，方法这是管理的科学性一面，但管理还要面对人，而人的思想是千变万化的，要选择一种他能接受的方式去沟通，这就需要管理艺术。一个团队需要的这种管理艺术越少越好，如果每一个人都能直面事实，不要考虑“面子”，个人利益，为什么还要艺术呢？所以“直面事实”是我们的终极追求。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >管理靠流程，尤其是关键流程不能省，我不止一次的碰到科学规律带给我们的惩罚，一个产品从研发到市场，要走过的路，恰似婴儿到成人。我们能做的只是少走弯路，我们不可能跨越某个阶段。当我们没有把试生产的问题都解决，当我们没有把该测的项目都测过，以侥幸心理对待，等待我们的结果往往是“欲速则不达”。当然，管理者要分析判断的是针对一个产品的状况，那些是“关键流程”以及如果要跨越某个阶段的风险评估。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >管理靠细致，对作业面的所有工程师“心细如发”可能是共同的个性特质要求。硬件工程师在<span lang="EN-US"   >Debug</span>一片板的时候，最基本的是先看有无连焊，虚焊，漏焊和错焊，这需要的就是心静心细。测试工程师在观察，描述一个<span lang="EN-US"   >Bug</span>时，心细也是必要条件。因为有的工程师在写测试报告时经常丢三落四，特别是把“……不能<span lang="EN-US"   ><span style=""   >&nbsp; </span>Pass</span>”，漏写成“……能<span lang="EN-US"   >pas</span>”
分析下来，也并不是不负责任，而是心粗。为此我曾对<span lang="EN-US"   >2</span>位粗心的工程师做过一种培训，就是每天花一小时，让他们把一碗黑白混合的芝麻分开，开始几天分开的芝麻总有混杂，尤其是会混杂半粒的芝麻，经<span lang="EN-US"   >2</span>周的时间，才真正半粒也不混杂。为了锻炼心静，我们还举行用筷子同时夹起三粒生花生米的训练。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >管理靠方法，才能少出错误，我们的软件工程师有时一天<span lang="EN-US"   >update</span>几次程序，可往往最后的一次更改，不是在上一次的程序上，错改到上上次的程序上，这是缺少版本号的管理。借助一些规范化的表格，比如设计文件<span lang="EN-US"   >List</span>，<span lang="EN-US"   >Debug</span>分析纪录<span lang="EN-US"   >List</span>，试生产<span lang="EN-US"   >Check List</span>，测试项目<span lang="EN-US"   >List</span>，测试表格等也会保证所做工作不被遗漏。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >人的天性容易趋利避害、避重就轻、文过饰非，这是人的心理决定的。尤其是做项目时，心存杂念，一心二用，出差错那是必然的。所以在研发项目中<span lang="EN-US"   >check</span>机制的建立是必要的。检查者不是全部重复设计者的工作。重要的是要将全部设计环节中的要点找出。要在其工作输出的重点上检查，这正如铁路巡道工，他在漫长的铁轨上主要是检查铁轨的结合处的螺栓松动与否，并不是等效的在每一米铁轨上平均花时间。根据不同情况，检查时这几点可供参考：要用与执行者不同的方法进行核算；进行试验<span lang="EN-US"   >/</span>测试确认；进行新设计与已有成熟设计的类比；对设计文件的审查特别要注意与设计实物的相符合；要设立一些简单易行的验证方法；检查者要做文字记录并保存；检查者要和设计者进行良好的人际沟通，要充分了解其设计思路。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >研发工程师的工作特性是需要安静少被打扰，以利于他的思考；而且工程师又往往爱面子——虽然这不见得是对的。因此借助网络的管理是很好的方法，因为透过网络传递信息，过滤了人的情绪化，而且文字有追溯性。除了<span lang="EN-US"   >Email</span>，我们用了<span lang="EN-US"   > TUTOS</span>系统来实时管理研发项目中发生的问题和传递信息。这实际上是一个类似<span lang="EN-US"   >BBS</span>一样的网络软件，只是具有更多的管理功能，如按项目设置成员和权限，问题目前是处理状态还是已解决状态，并且任何人发布新信息时，<span lang="EN-US"   >TUTOS</span>系统会有<span lang="EN-US"   >mail</span>自动发给相关成员，提示去<span lang="EN-US"   >TUTOS</span>系统看。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在各种研发电子文件的管理上，先是做好了科学的分类，并且有专人来定期整理和更新，当资料越来越多，后来又考虑开发像搜索引擎一样方便的搜索功能，这样可以大大方便、有效利用现有资料，只惜这件事没做完。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >对不同层次的研发工程师需要不同的管理，对有项目经验的工程师我基本上是做目标管理，仅看结果；对新手则要更多的关注过程，否则也许就会“翻船”。<span lang="EN-US"   >.</span>我对项目管理的成败判定标准是：设计一块主板，如果出现了原理性的错误；或者如果<span lang="EN-US"   >schedule</span>延迟了<span lang="EN-US"   >10</span>天以上，那一定是管理问题，而不是设计者的技术问题。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >对不同专业的研发工程师需要不同的管理，比如对测试工程师，他们工作中对创新要求并不高，更重要的可能是细心和逻辑分析力。我给测试工程师<span lang="EN-US"   >3</span>个目标：第<span lang="EN-US"   > 1 </span>个目标是能够按时并一次将被测主板存在的问题都测出来；第<span lang="EN-US"   >2</span>个目标是能够对测出的问题做原因分析；第<span lang="EN-US"   >3</span>个目标是对测出的问题给出解决方案。完全达到这<span lang="EN-US"   > 3</span>个目标，可能他需要在这个专业上做<span lang="EN-US"   >8</span>年以上。同时为了让测试工程师知道自己处于何水准，我们设计了<span lang="EN-US"   >2</span>个考核指针：用每一测试项目所花时间与标准测试时间之比来考核其工作效率；用一次<span lang="EN-US"   >bug</span>测出率来考核其工作质量<span lang="EN-US"   >(</span>这个指针得出，需要该产品后续的测试结果，故不是实时考核指针。<span lang="EN-US"   >) </span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们曾经做过全年的统计，在研发阶段和量产阶段对那些看表面现象是技术造成的问题做分析，结果令大家都很吃惊的是有<span lang="EN-US"   >70%</span>的问题是在管理环节可以避免的，只有<span lang="EN-US"   >30%</span>的问题确实是当时对技术掌握不够造成。我最近接触了一些国内<span lang="EN-US"   >IT</span>公司的总裁，发现真正认识到研发中心缺管理的不多，实际上是国内<span lang="EN-US"   >IT</span>公司研发中心不仅缺技术同样缺管理。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >13. </span><span style=""   >技术传承 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >当没有其它激励时，几乎任何一个人都是，当他总是在向别人分享他的知识，而得不到别人的知识反馈，慢慢地他会停止这个行为 。所以技术传承要做的好，就要保证技术双向传播，技术共享，各取所需，共同提高。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >因为新人多，有经验的工程师少，中心采用了两种培训模式。一种是将研发<span lang="EN-US"   >PC</span>主板的硬件技术分成十几个专题，每个专题的题目一般都定得很具体，界定范围，避免泛泛而谈，力争将一个小问题讲深讲透，能够对设计有指导作用，确定题目和内容主要由我和几个有经验的工程师来做。包括新人每人研究一两项，研究时间是利用做项目以外的时间，花<span lang="EN-US"   >2~3</span>个月的时间。然后每人轮流讲给大家，会前<span lang="EN-US"   >2</span>天将内容发给大家。会议形式是讨论会，会上鼓励提问题，充分交流，报告者自己也可提出自己没有理解的部分的问题。一轮讲完后，再来有延续性的第<span lang="EN-US"   >2</span>轮。几年下来，每人都成为某个分支的一技之长者，大家也花较少的时间学到了较多的技术。一种是师徒制，这是传统的方法但也是有效的方法，同时徒弟的进步也纳入双方的绩效考核。徒弟技术能力不够，总是在请教师傅，他怎么样使别人一直愿意教他？他可以帮师傅分担一些数据收集，整理以及在过程中的一般测试、试验。让师傅去把精力放在技术深层次问题分析和思考上，以加速问题的解决。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >一个工程师应有技术专长，但更重要的是一个工程师要想具备一技之长，应该有一些基本能力。分析能力就是其一。怎样养成面对问题先分析的习惯？有这样一件事：中心规划了要开发一个加密软件，作为主板的附件。我记起几年前看过的《王选文集》其中的“软件规划”，内容是我迄今为止最有实际指导价值的。我就请我们的一位同事去买，也特别对他说，重点是要看其中的“软件规划”。他是复旦的应届毕业生，跑了一趟上海书城和科技书店，就说买不到，就说没有办法。我当时说给你<span lang="EN-US"   >3</span>天时间，买到后再来上班，<span lang="EN-US"   >3</span>天都买不到，你就不用来公司了，他委屈的要哭。结果在同事的鼓励下，还是又去想办法了。结果第<span lang="EN-US"   >2</span>天就买到了。我相信这件事对他刺激是很大的。这里面就存在着对一个任务“分析”的问题<span lang="EN-US"   >(</span>当然还包含着做事的观念<span lang="EN-US"   >)</span>。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >为什么要买这本书？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >有几种方法可买到此书？（应该有<span lang="EN-US"   >10 </span>种以上方法，你能想到几种呢？）
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >各种方法需要的时间为多少？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >各种方法钱的花费为多少？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >对各种方法，先用哪种，后用哪种？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >哪种方法综合效果最好？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >* </span><span style=""   >当买不到书时，有何变通方法？ <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >这是一件工作中的小事，但良好的习惯靠平时养成。做任何事，当你能够理性地去进行分析，往往就成功了一半，相反，则埋下了失败的因子。有时，即使是成功了，也是一种运气而已。再举例。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >在量产的产品中，我们多次碰到这样一个问题：对于主板上的某一个<span lang="EN-US"   >chip</span>，有些生产日期的产品在主板做高负荷的测试时，会有批量不良，分析发现是因为这颗<span lang="EN-US"   ><span style=""   >&nbsp; </span>chip</span>在不同生产日期，因其制程的微小差别和电子元器件本身的“温飘”，造成住板上某个输入或输出的信号偏差超出容许范围。对这类情况，有人认为这在设计阶段不可能被发现的，所以要在产品生产后做比较多的做高负荷测试以后，才会抓出这些问题。应该是这样吗？硬件工程师在设计时，是不是应该考虑到<span lang="EN-US"   > chip </span>参数的上<span lang="EN-US"   >/</span>下限的极端状况呢？在承认书中是不是应该明确<span lang="EN-US"   >chip </span>参数允差范围？研发部门做测试时，是不是应该做各种在承认书允差范围内的测试？这样都做到了，设计余量足够，问题就会少发生，只会发生超出承认书允差的不良，而这种不良是应由<span lang="EN-US"   >Chip </span>供货商<span lang="EN-US"   >/</span>制造商来承担的，这可以在签订商务合同时来约束。经过这样的分析，我们可以有这样的结论：设计者应充分考虑<span lang="EN-US"   >Chip </span>参数偏差的“上<span lang="EN-US"   >/</span>下限”时线路的情况，设计端的测试部分应能作参数拉偏极限试验。量产时是不应做这样的测试的，否则问题暴露在制造端，花费的成本就太高了。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >类似这样的事例，我们都把它做成了案例，不仅现在用，也作为今后培训新人的教材，因为这些事情绝不是发生在<span lang="EN-US"   >1</span>～<span lang="EN-US"   >2</span>个人身上。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >不论对于哪一类专业的工程师，分析能力都是至关重要的。多角度的思维，特别要有逻辑的去分析：要细致地观察现象，按事物发展的客观实际，立论有据，推论合理，不能跳过问题去下结论。要循序渐进，由表及里，不是浅尝辄止。如<span lang="EN-US"   >1 </span>块<span lang="EN-US"   >VGA </span>卡测试时发现花屏，换了<span lang="EN-US"   >memory
</span>就好了，问题就这样结束了吗？远远的没有，要清晰地描述发生问题的现象：如所用配置，操作的动作，屏幕显示；再去分析真因是原材，设计还是制程带来的问题？针对原因要有对策，现在怎么办？今后怎样预防，怎样根除？
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >有没有分析能力是判断一个低阶技术人员和中阶技术人员的分水岭。分析能力的强和弱是中高级人才的判断标准之一。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >“授人以鱼，不如授人以渔”一个成人<span lang="EN-US"   >IQ(</span>智商<span lang="EN-US"   >)</span>是基本不能提升的，而<span lang="EN-US"   >EQ(</span>情商：综合的分析<span lang="EN-US"   >/</span>应变能力<span lang="EN-US"   >)</span>是可以锻炼培养的。我们在传授技术的同时更注重传授方法。把追求“<span lang="EN-US"   >Know-how</span>”变成一种习惯。这也是很多新手飞快进步的原因。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >除了基础知识的补充，基本技能的培训，我们还尽量的争取让新手早日介入项目设计，这正如在球场上运动员在运动中才能进球。在设计中碰到问题并解决问题，那种印象要比听别人讲深得多。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >我后来还兼管了工厂的工程部，用这套办法稳定了技术队伍并由此吸引了很多周边同行厂的工程师来加盟。那些在别的公司互相提防，技术封闭的环境下工作的同事，到了我们这种开放的环境，激发了很大的学习和工作热情。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >14. </span><span style=""   >怎样留人 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >人在不同的层次要求是不一样的。<span lang="EN-US"   >1999</span>年在一个<span lang="EN-US"   >IT</span>年会上，王选教授说，他当时开发激光照排技术的动力是看到排字工人那么辛苦，有一股要把他们从繁重的体力劳动中解放出来冲动。他说从来没想到有奖金啊荣誉什么的，他还说”市场的成功才是对研发人员的最大激励”。这一点之所以记得这么牢，是自己十几年前也是这么想。那时没有生存的压力，又受了那么多年的教育。甘于清贫，乐在奉献是那几代知识分子的行为准则。现在说这些，连自己都觉得是在唱高调。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >自我需求金子塔理论马斯洛<span lang="EN-US"   >Abraham Maslow </span>提出了人有生理，安全，社会，尊重，自我需求这些由<span lang="EN-US"   > a </span>至<span lang="EN-US"   > e </span>由低级至高级的全面需求，它形同金子塔。这些需求对人来说是逐步的分阶段的
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >a) </span><span style=""   >生理需求包括食物温暖住所； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >b) </span><span style=""   >安全需求稳定的环境工作人身安全； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >c) </span><span style=""   >社会需求亲情友情爱情； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >d) </span><span style=""   >尊重需求自我意识增强需要别人的尊重； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >e) </span><span style=""   >自我需求要求自我实现有独立的精神和人格对事物有深刻的认识力。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >自我需求就是一种最好的激励。怎样使得大家有工作激情，我的体会是首先要从招聘开始。多年的经验表明。如果一个人做他喜欢做的事，可以省掉很多管理上的要求。他的进步也会很快。我的副手李兴中就是，从中学开始就喜欢电子和计算机有很多知识和动手能力是在工作前就具备了。上班在做设计，下了班休息一下，大部分时间还是在研究计算机技术，工作的内容就是他的爱好。所以选择“做我所爱，爱我所做”的人是最合适的。可是，在目前的社会条件下，大多数人还是为了生存，或者说是为了生存的好一点，才来做<span lang="EN-US"   >IT</span>。象本人就是，完全就是为了家庭和孩子能够有一个良好的生活水准。虽然也很勤奋的工作，但我的感觉和李兴中是不一样的，我是靠对一种家庭和职业的责任在支撑。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >对处于不同需求阶段的工程师首先要对他们心态的了解。比如对从农村来的应届生，他可能最需要的就是一份不错的工资和稳定的工作，他不怕辛苦。而上海本地的应届生他们没有那么大的生存压力，可能不愿意工作太辛苦。在工作安排时要做相应的考虑，前提当然是，一份耕耘，一份收获。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   >对已在其它台资公司做过几年的工程师他们可能最需要的是做项目标机会。我们就要尽可能给他这个机会，否则就会留不住人。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >对资深的工程师来说，就要有技术上继续进步的机会或者向管理位置发展的机会。我在对技术骨干的态度和做法可能是我做成一些事和他们愿意长期跟我的原因。我的做法是看到有潜质的工程师会专门去有意识的去观察和给他机会锻炼。我往往会破格提拔，以给其大剞量的刺激，激发其潜质。这方面我是做的比较成功的。在同创时从维修工程师一下提拔为测试部的主管的李兴中；在深圳海尔科技将优秀软件工程师宋雅松的工资提的比我还高，让海尔总部领导也说，从没一个部门领导能够为员工做到这个状况，还有在我兼管<span lang="EN-US"   >ABIT</span>工厂工程部时发现的<span lang="EN-US"   >PE</span>顾峰源。远远不止这几位，事实上这些人确实是出类拔萃的，不论在那家公司都会脱颖而出。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >为了大家对工作对技术有热情，在开技术研讨会时，为鼓励发言，我们采用发言计分，纳入日常绩效考核；我们还自编了随机抽取发言人的小软件，以保证公平性。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我们也制定了不同级别的职务说明书，上面列清楚了要掌握的专业知识<span lang="EN-US"   >/</span>技能，做项目的能力，语言表达及写作能力等具体的要求。让大家有明确的努力目标。特别是坚持不论资排辈，而是看每人的绩效确定奖励。在这样一种环境下，一批人得到了个人能力的大幅提升。<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   >对成熟工程师我经常提一些需将技术知识贯穿起来的问题，以激励他们把技术融会贯通。象“在计算机上当你按下一个‘<span lang="EN-US"   >A</span>’字，键盘有了什么反应，信号怎样传给主板上的南桥，<span lang="EN-US"   >CPU</span>又起了什么作用？显卡怎么样接受信号传给显示器”？“为什么在主板不能开机时，既能通过改电源电路硬件解决，也能不改电路硬件而通过改<span lang="EN-US"   >BIOS</span>相关参数解决”？还有“为什么<span lang="EN-US"   >PC</span>有<span lang="EN-US"   >BIOS</span>，<span lang="EN-US"   >OS</span>，<span lang="EN-US"   >Driver</span>，<span lang="EN-US"   >AP 4</span>类软件？为什么<span lang="EN-US"   >TV</span>没有？” <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >在珠三角，长三角这两块热土，就外部来说有着大量的就业机会，就内部来说对工程师们公司种种的不适应，造成工程师们的频繁流动，其结果于自己于公司大都不利。我在这<span lang="EN-US"   >2</span>个地方都工作多年，所管的研发团队都保持了优秀人材的相对稳定，总体流动比率大大低于同行平均水平。总结起来也觉得并不是很困难的事。<span lang="EN-US"   ></span></span></p>

<p><span style=""   >第一，一份中等或以上水准的薪水是首要条件，毕竟大多数工程师没有达到“住”无忧的经济水平；<span lang="EN-US"   ></span></span></p>

<p><span style=""   >第二，有想做的事可做； <span lang="EN-US"   ></span></span></p>

<p><span style=""   >第三，有技术可学，毕竟市场竞争激烈；<span lang="EN-US"   ></span></span></p>

<p><span style=""   >第四，所在的部门公平，人际关系简单。<span lang="EN-US"   ></span></span></p>

<p><span style=""   >如果再提一点高要求，那就是个人有发展空间，公司有发展前景。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >15. </span><span style=""   >两岸工程师 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >我接触过一些台湾中年工程师，他们谦逊，内敛，专业，礼貌，心态平和。和大陆台湾中年工程师比较，两者对工作的观念方法，处事原则相差不大。而以青年工程师比较，大陆和台湾两者对工作的观念方法相差很大。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >目前中国处于社会变革的时期，整个社会显得浮躁。大陆的青年工程师已极少有人愿意“十年磨一剑”。往往做了一年项目助手，第二年就想做主要角色，第三年就要做<span lang="EN-US"   >Leader</span>。而且一旦达不到个人目标就频繁跳槽，致使成不了真正的技术精深的专家。而台湾的青年工程师即便是一个普通的测试职位也往往愿意一做十年不改行，成为专家级的工程师。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >大陆工程师频繁跳槽最主要的表象原因是外部机会多。其次就是薪水低，学不到技术，公司环境不好。事实上频繁跳槽，在上个世纪的<span lang="EN-US"   >80~90</span>年代台湾也是这样的状况。最根本的原因是经济上的压力，眼看着房价节节上升，十年不吃不喝也买不起房，这时应遵守的职场伦理就被丢在了脑后。而所在的公司因为前期投入的培训费还没收回，薪资自然是慢慢的加。这样跳槽就在所难免，“存在就是合理”既然他到外面公司能有大幅度的提薪，为什么原公司就不能做到呢？可能就是这口<span lang="EN-US"   ><span style=""   >&nbsp; </span></span>“气”咽不下去，结果多数还是双方受损，所以如果大环境不变这是公司和个人还继续要承担的阵痛。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

http://www.csdn.net/ </span></span></p>

<p><span style=""   >我们往往要的是一个好结果，这是没错的，但好的工作输出来源于好的工作输入，这常被人忽略。软件行业有一句著名的格言：“进来的是垃圾，出去的还是垃圾”，把好输入关是极其重要的。所以特别把<span lang="EN-US"   >3</span>年前在我们中心杂志我写的一篇文章贴在这里。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >*</span><span style=""   >敏感性分析（技术、市场、竞争对手变化带来的风险，如何抵御？战争、自然灾害等不可抗拒力带来的风险）； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.2.5</span><span style=""   >法规法律的可行性 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >符合法规、法律、环保规范，符合销售市场的民俗、宗教习惯。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.3</span><span style=""   >制定产品标准或<span lang="EN-US"   >SPEC </span></span></p>

<p><span style=""   >按照国家<span lang="EN-US"   >/</span>国际<span lang="EN-US"   >/</span>行业的相关标准制定自己的产品标准；<span lang="EN-US"   >IT</span>行业有其技术、产品更新快的特点，往往很难制定产品标准，这时可以根据用户需求和可行性分析报告，制定<span lang="EN-US"   >SPEC</span>，包括功能和性能，还应包括包装、环境、<span lang="EN-US"   >MTBF</span>、安全、环保及引用的标准、规范。<span lang="EN-US"   > SPEC</span>要用定制的表格形式；特别强调要经用户签字确认。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4</span><span style=""   >用户说明书 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >用户说明书要用非专门术语的语言，充分地描述其具有的功能及基本的使用方法；并能够说明特定的情况下的应对方法和注意事项。应特别注意下列各点： <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.1</span><span style=""   >用户说明书应在设计产品前就写，而且是尽可能的完善，这是成功研发产品得到的宝贵经验。特别是对未设计过的新一类产品，在设计产品前就写，是至关重要的（否则极易设计出不符合用户需求的想当然产品）；
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.2</span><span style=""   >站在用户的角度，根据<span lang="EN-US"   >SPEC</span>和用户需求编写用户说明书，一份好的说明书应易读易用，让最普通水平的用户读后都能够正确、快速、无误操作的使用；
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.3</span><span style=""   >对一事物的描述，人易接收的程度依次为三维动画，二维动画，图片，表格，公式，文字（有语音辅助更好）。编写用户说明书要充分考虑这个规律；
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.4</span><span style=""   >轻重要分开，逻辑要清楚。如对应用软件的使用说明要明确的表叙：用户应进行的实际操作；屏幕有那些反应；有什么要注意的事项。对这些不同的内容，可用不同的字体、颜色、段落来区分；
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.5</span><span style=""   >当说明书内容很多时，可分为基本功能和高级功能，前几页是基本功能，让用户可迅速使用； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.6</span><span style=""   >说明书的外观、格式应符合企业的<span lang="EN-US"   >CI </span>形象； <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.4.7</span><span style=""   >说明书应在产品发表时和在研发过程中必要时更新，一般来说，变更越少，体现产品规划越高。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.5</span><span style=""   >项目任务书 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >除了在规划阶段（<span lang="EN-US"   >1.1</span>—<span lang="EN-US"   >1.4</span>的内容和<span lang="EN-US"   >1.6</span>的内容）给产品规划人员下达的上述内容任务书，这里的项目任务书是指经确认，可立项设计的项目； <span lang="EN-US"   ></span></span></p>

<p><span style=""   >项目任务书内容包括项目名称、设计组成员、启动和结束（可分几大阶段）的间。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.6</span><span style=""   >制定<span lang="EN-US"   >Schedule </span></span></p>

<p><span style=""   lang="EN-US"   >Schedule</span><span style=""   >的三要素是人、事、时间，要制定完整的具体的，直到<span lang="EN-US"   >MVT</span>阶段设计输出的项目<span lang="EN-US"   >Schedule</span>。并在执行中适时更新，以符合当前状况。要借助于项目管理软件，以实现科学规范的管理。；<span lang="EN-US"   >Microsoft Project </span>是一种功能强大而灵活的项目管理工具，可以用于控制简单或复杂的项目。它通过各种图表能够安排和追踪所有的活动，从而对进度了如指掌。而且现在，通过安装在公司的<span lang="EN-US"   > intranet </span>或<span lang="EN-US"   > Internet </span>上的<span lang="EN-US"   ><span style=""   >&nbsp; </span>Microsoft Project Central </span></span></p>

<p><span style=""   >，可以比以往更为方便地与有关人员交换项目信息。如它可以实现子项目的关联性；子项目的重要等级等等。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >1.7 </span><span style=""   >特殊性 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >以上是一个通常项目的设计输入，对特定的项目会有些不同，如有的软件项目需要“软件需求分析”（不是指“用户”）。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >2.</span><span style=""   >设计输入的评审 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >计算机软件行业有一句著名的格言：“进来的是垃圾，出去的还是垃圾”把好输入关是极其重要的。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >2.1</span><span style=""   >设计输入评审的参加人员 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >设计输入的评审应有用户（或代表）、管理者、产品规划工程师、产品设计师、测试工程师、采购、销售人员参加。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >2.2</span><span style=""   >设计输入评审的要求 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >对设计输入的内容应分别进行评审，不完整的、含糊的和矛盾的要求应予以解决，没有遗留问题，再进行下一步的工作。这也是<span lang="EN-US"   >ISO9000:2000</span>版的要求
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >2.3</span><span style=""   >设计输入的存档 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >应将确定的设计输入按文件管制的规定存档并保持。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >18. </span><span style=""   >扬长避短 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >除非经历一个人生大劫，否则人的个性特质是基本不会改变的。有时看起来是改变，那可能只是因为外界的压力，暂时的收敛。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >象我的个性是属于面对不合理的事是“疾恶如仇”，说话是直面事实，不怕得罪人；做事是追求完美，一追到底，风风火火，速战速决。象我这类人，在处理一些事的时候，一定有可取之处。如果要说负面的那就是易“得罪人”，工作要求高，给别人较大压力。我处理事时，他第一次出差错，我会和颜悦色地让他注意，第二出同样的差错，我会言词严肃，第三次还出同样的差错，我会忍不住严辞厉色。“对事不对人”作为一种管理理念是对的，但事是人做的，不把做事的方法说给他，不把责任心强调，不分析为什么多次出现这样的问题说给他听，怎么可以？这样对了解我的人而他又有一定承受力，没什么不良后果。但对脆弱的人往往是受不了。我承认在任何情况下都应该和颜悦色，但做不到，我仔细想过我为什么这样，原因应该是两条，一是个性使然，一是强烈的责任心。两者去掉任一条，都不会象我“那么‘顶真’，象做自己的公司一样”，这是我的同事经常说我的话。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >而且我自己发现，我一激动，思维会特别敏捷，前后事情会在脑子里面全部激活起来。 <span lang="EN-US"   ></span></span></p>

<p><span style=""   >对多次出同样的差错的人我做过分析，一个是责任心不够；一个是做事的方法或技术没掌握；还有一个不被人注意的原因是，他的“联想能力”能力不够，我发现一些人，确实不是前两个原因出错。主要是他脑子里面不记事，上周的事他记不住，面对眼前的事也不会作相关联想，这样的个性特质往往也不具备并行作业的能力。<span lang="EN-US"   ><span style=""   >&nbsp; </span></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   lang="EN-US"   ><span style=""   >&nbsp;</span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >实际上，一贯的“和风细雨”式的管理和平时“和风细雨”加上偶尔地“暴风骤雨”式管理，都是一种管理风格，没有对错之分，只是一种选择，因为各种管理风格都有大量的成功案例。
<span lang="EN-US"   ></span></span></p>

<p><span style=""   lang="EN-US"   >&nbsp;</span></p>

<p><span style=""   >不同特质的人，应该选择做不同的事。有的人爱说话，善于口头表达，也许做<span lang="EN-US"   >FAE</span>合适；有的人心细，观察力强，也许做测试工程师合适；有的人心静，做的住，也许做<span lang="EN-US"   >PCB layout</span>工程师合适；有的人富于创意，善于“自来熟”也许做<span lang="EN-US"   >PM</span>合适；有的人富于创意，善于“自来熟”也许做<span lang="EN-US"   >PM</span>合适 “江山易改，本性难移”也许硬着头皮改了个性，他的优点也随之消失。与其化大量时间去修身养性，不如扬长避短去做符合个性特质的事。
<span lang="EN-US"   ></span></span></p>
